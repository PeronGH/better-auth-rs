---
title: Database
description: SeaORM-backed database integration for Better Auth.
---

Better Auth uses a SeaORM `DatabaseConnection` for auth persistence and re-exports the SeaORM runtime directly from the root crate.

The supported public path is:

1. Create a `DatabaseConnection`.
2. Run the built-in auth migrations.
3. Pass the connection to `BetterAuth::new(config).database(...)`.

## Setup

```rust
use better_auth::{run_migrations, AuthConfig, BetterAuth, Database};
use better_auth::plugins::EmailPasswordPlugin;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let database = Database::connect("sqlite::memory:").await?;
    run_migrations(&database).await?;

    let auth = BetterAuth::new(
        AuthConfig::new("your-very-secure-secret-key-at-least-32-chars-long")
            .base_url("http://localhost:3000"),
    )
    .database(database)
    .plugin(EmailPasswordPlugin::new().enable_signup(true))
    .build()
    .await?;

    println!("plugins: {:?}", auth.plugin_names());
    Ok(())
}
```

## PostgreSQL

For production, pass a PostgreSQL connection string to SeaORM and then run the same migrations:

```rust
use better_auth::{run_migrations, Database};

let database = Database::connect("postgresql://user:pass@localhost:5432/mydb").await?;
run_migrations(&database).await?;
```

## Testing

Repo-local tests use SQLite through the same `DatabaseConnection` API plus `run_migrations`.
The dual-server and client SDK compatibility checks still validate behavior against the TypeScript
reference server.

## Notes

- `run_migrations` uses the default `better_auth_migrations` migration table.
- `run_migrations_with(&db, MigrationOptions { table_name, .. })` lets you choose a custom auth migration table.
- `auth_migrations()` exposes the built-in Better Auth migrations for composition with an application migrator.
- `BetterAuth::database_connection()` is the supported way to reuse the exact shared SeaORM connection in application code.
- `MemoryCacheAdapter` is still available for cache/rate-limit use cases. It is separate from database persistence.
